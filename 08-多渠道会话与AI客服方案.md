# 08 · 多渠道会话与 AI 客服方案

最后更新：2026-07-06

---

## 一句话说明

未来目标不是把 Twenty CRM 改造成完整聊天系统，而是用 **Chatwoot** 统一接入 WhatsApp、官网聊天、Email、社媒等多方渠道，再通过一个中间层把客户资料、聊天摘要、来源、会话链接等稳定写入 **Twenty CRM**。

对用户前端体验的目标是：**销售尽量在 CRM 的联系人页面里查看多渠道联系人、聊天记录、附件，并能直接回复；底层实际消息收发和完整记录仍由 Chatwoot 承担。**

---

## 最终目标

公司希望把客户在多个平台上的沟通记录统一沉淀，销售和管理者能在 CRM 中搜索客户、需求、产品兴趣和沟通摘要，并能点开查看完整聊天上下文。

目标包括：

- 多方渠道统一接入：WhatsApp、官网聊天、Email、Instagram、Facebook、Telegram、Line、LinkedIn、Slack 等
- 销售统一处理客户消息，减少多平台来回切换
- 在 CRM 联系人页面中呈现多渠道联系人身份、会话列表、详细聊天记录和回复入口
- 客户资料、来源、感兴趣产品、沟通摘要写入 CRM
- 图片、视频、文字、链接、附件等完整聊天记录可追溯
- 官网表单自动写入 CRM
- 官网 AI 客服可回答基础/产品问题，答不了时转人工
- 系统在网络波动时能避免漏写、重复写、错写

---

## 推荐总架构

```text
多方渠道
WhatsApp / 官网聊天 / Email / Instagram / Facebook / Telegram / Line / LinkedIn / Slack / 其他
        ↓
Chatwoot
统一收消息、发消息、保存上下文、管理坐席、保存附件
        ↓
中间层 Sync + AI Service
去重、重试、字段映射、AI 判断、知识库检索、摘要生成
        ↓
Twenty CRM
客户、公司、商机、任务、来源、摘要、会话链接、内嵌会话面板
```

核心原则：

1. **Chatwoot 做聊天中心**
2. **Twenty 做 CRM 主数据**
3. **中间层做唯一写入 CRM 的出口**
4. **完整聊天记录留在 Chatwoot，CRM 保存摘要、索引和前端呈现入口**
5. **用户可以在 CRM 里看和发，但底层仍通过 Chatwoot 收发**

---

## 各系统分工

| 系统 | 负责什么 | 不负责什么 |
|---|---|---|
| Chatwoot | 多渠道聊天、人工坐席、会话上下文、附件、会话状态、消息收发 | CRM 主数据、商机管理、长期客户画像 |
| Twenty CRM | 联系人、公司、商机、任务、客户来源、产品兴趣、AI 摘要、会话链接、联系人页内嵌会话入口 | 直接实现 WhatsApp/LinkedIn/Slack 等渠道协议 |
| 中间层 | API 接收、webhook 处理、去重、重试、字段映射、AI 分流、知识库检索、摘要写入、CRM 会话面板代理接口 | 直接替代 Chatwoot 或 Twenty |
| 官网 | 表单、官网聊天入口、展示 AI 客服入口 | 保存 CRM 密钥、直接写数据库 |

---

## 为什么不建议把 Twenty 改成完整聊天系统

Twenty 是 CRM，不是在线客服/即时通讯系统。以下能力更适合放在 Chatwoot：

- 实时收发消息
- WhatsApp 接入
- 官网在线客服窗口
- 人工坐席接管
- 未读消息、会话状态、分配规则
- 图片、视频、文件附件
- 聊天上下文和历史记录
- 多销售协作

Twenty 中应该保存和呈现：

- 客户是谁
- 来自哪个渠道
- 对什么产品感兴趣
- 最近聊了什么
- 当前是否需要跟进
- 对应 Chatwoot 会话链接
- 多渠道身份
- 会话列表
- 聊天摘要
- 内嵌聊天记录面板

推荐目标不是“销售必须跳到 Chatwoot 才能聊天”，而是：

```text
销售看到的是 Twenty CRM 联系人页面
        ↓
联系人页面内嵌一个“多渠道会话”面板
        ↓
面板通过中间层读取 Chatwoot 消息
        ↓
销售在 CRM 里输入回复
        ↓
中间层调用 Chatwoot API 发出消息
```

这样对用户来说像是在 CRM 里沟通和回看；对系统来说，消息事实来源仍是 Chatwoot。

## CRM 内呈现聊天的三种方案

### 方案 A：低成本跳转方案

Twenty 里只显示：

- 多渠道联系人身份
- 最近聊天摘要
- 最近沟通时间
- Chatwoot 会话链接
- 关键聊天摘要
- 附件链接

用户点击链接跳转到 Chatwoot 看完整会话和回复。

优点：

- 最快上线
- 改动少
- 风险低
- 对 Twenty 侵入最小

缺点：

- 用户会明显感觉在两个系统之间切换
- 不满足“前端呈现上在 CRM 中沟通”的最终体验

适合作为第一阶段过渡方案。

### 方案 B：CRM 内嵌会话面板（推荐目标方案）

在 Twenty 联系人详情页增加一个“多渠道会话”Tab 或 Widget。

示意：

```text
联系人详情页
├── 基本信息
├── 商机
├── 邮件
└── 多渠道会话
    ├── WhatsApp
    ├── 官网 chatbot
    ├── LinkedIn
    ├── Slack
    ├── 历史会话列表
    ├── 当前会话详情
    ├── 图片/视频/附件
    └── 回复输入框
```

工作方式：

```text
Twenty 联系人页
  → 读取当前 contact 的 chatwoot_contact_id
  → 请求中间层
  → 中间层调用 Chatwoot API 拉取会话和消息
  → CRM 页面展示聊天记录、附件、链接
  → 销售在 CRM 中回复
  → 中间层调用 Chatwoot API 发送消息
  → Chatwoot 负责真正发到 WhatsApp/官网/其他渠道
```

优点：

- 用户主要停留在 CRM
- 完整记录仍保存在 Chatwoot
- 不需要把 Twenty 改造成完整聊天系统
- 后续可以逐步扩展附件、搜索、标签、AI 摘要

缺点：

- 需要开发 Twenty 自定义前端组件或改 Twenty 前端
- 需要中间层提供稳定的会话查询、发送、附件代理接口
- 需要处理权限、身份映射、消息状态和错误提示

这是目前推荐的最终目标方案。

### 方案 C：直接把 Twenty 改成完整聊天系统（不推荐）

把消息协议、会话列表、附件、发送、未读、坐席、分配、多渠道适配等全部做进 Twenty。

优点：

- 理论上体验最统一

缺点：

- 开发和维护成本最高
- 会破坏 Twenty 升级路径
- WhatsApp、LinkedIn、Slack 等渠道协议都要自己长期维护
- 附件、消息状态、失败重试、坐席协作会变成重工程
- Chatwoot 的价值会被削弱

结论：不建议。

## CRM 内嵌会话面板的推荐数据流

### 查看聊天记录

```text
用户打开 Twenty 联系人详情页
        ↓
CRM 会话面板读取联系人绑定信息
        ↓
请求中间层：/crm/contacts/{id}/conversations
        ↓
中间层查询 Twenty 绑定字段 + Chatwoot API
        ↓
返回会话列表、消息、附件链接、发送者、时间
        ↓
CRM 页面渲染完整上下文
```

### 发送消息

```text
销售在 CRM 会话面板输入消息
        ↓
请求中间层：/crm/conversations/{id}/messages
        ↓
中间层校验销售权限和联系人绑定
        ↓
调用 Chatwoot API 创建 outgoing message
        ↓
Chatwoot 发送到对应渠道
        ↓
webhook 回流给中间层
        ↓
中间层更新 Twenty 摘要、最后沟通时间、搜索索引
```

### 查看附件

```text
Chatwoot 保存图片/视频/文件附件
        ↓
CRM 面板展示附件缩略图、文件名、类型、大小、打开链接
        ↓
需要权限控制时，由中间层生成短期可访问链接或代理下载
```

## Chatbot 的当前定义

这里的 chatbot 不是单指 AI，也不是单指机器人客服。

当前定义：

```text
chatbot = 一个承载工具 / 对话入口按钮
```

例如官网右下角的聊天按钮、WhatsApp Bot、API Bot，都可以视为 chatbot 入口。

用户点开 chatbot 后可以进行对话，但底层真正承接会话的是客服系统：

```text
用户点官网按钮
  → 打开聊天窗口
  → 消息进入 Chatwoot
  → 客服/销售/后续 AI 能力回复
  → 聊天记录留在 Chatwoot
  → CRM 内嵌面板可查看和回复
```

AI 客服只是后续接入的一层能力，不影响当前 Chatwoot + CRM 打通设计。

## CRM 中如何搜索聊天记录

推荐分三层实现。

### 第一层：摘要搜索（最推荐先做）

Twenty 中保存每次会话的摘要、标签、产品兴趣、客户诉求、最近消息时间和 Chatwoot 链接。

优点：

- 简单稳定
- CRM 数据不会爆炸
- 销售能快速搜索客户和需求
- 点链接可进入 Chatwoot 看完整上下文

### 第二层：关键记录同步

中间层把重要会话摘要写成 Twenty Notes，例如：

```text
2026-07-03 官网 AI 客服会话摘要：
客户来自沙特，关注某产品，询问 MOQ、价格、交期，已留下 WhatsApp。
完整记录：https://chatwoot.example.com/app/accounts/1/conversations/123
```

适合需要在 CRM 中追踪关键沟通历史，但不要求逐条消息全文检索的阶段。

### 第三层：全文搜索

如果必须搜索每一句聊天原文，建议加独立搜索库，例如 Meilisearch 或 Elasticsearch。

不建议把所有原始消息、图片、视频、附件完整复制进 Twenty。Twenty 不是为海量聊天消息和附件存储设计的。

如果采用“CRM 内嵌会话面板”，CRM 搜索可以分为两类：

- CRM 原生搜索：搜客户、公司、商机、摘要、标签、产品兴趣
- 会话全文搜索：由中间层或独立搜索库提供，在 CRM 会话面板内呈现结果

这样可以保证用户在 CRM 里搜索和回看，但不会把完整聊天存储压力转移给 Twenty。

## AI 板块定义

当前 AI 相关需求分为三类，但它们不是三个完全独立系统。

| 名称 | 正确定义 |
|---|---|
| Chatbot | 一个客户入口，例如官网聊天机器人、WhatsApp Bot、API Bot，也可以视为多方渠道之一 |
| 智能客服 | AI 判断器和执行器，负责判断客户问题该由 AI 回答还是转人工 |
| 知识库 | AI 回答问题时检索的公司资料、产品资料、FAQ、文档 |

推荐流程：

```text
客户从 Chatbot / WhatsApp / 官网聊天进入
        ↓
Chatwoot 收到消息
        ↓
AI Router 判断
        ↓
能回答：检索知识库后自动回复
不能回答：转人工坐席
        ↓
会话结束后生成摘要
        ↓
写入 Twenty CRM
```

## 官网如何接入

官网有两个明确入口。

### 1. 官网表单

表单字段包括客户基本信息、感兴趣产品等。

推荐流程：

```text
官网表单
  → 中间层 API
  → Twenty 创建/更新 People、Company、Opportunity
  → 可选：同步创建 Chatwoot Contact
```

注意：

- Twenty API Key 不能放在官网前端
- 官网只调用中间层
- 中间层负责字段校验、去重、写入 CRM

### 2. 官网 AI 客服

推荐流程：

```text
官网聊天窗口
  → Chatwoot
  → AI Router 判断
  → 知识库能答：AI 回复
  → 知识库不能答：转人工
  → Chatwoot webhook
  → 中间层
  → Twenty 更新客户资料、摘要、会话链接
```

人工什么时候介入、如何介入，可后续单独设计。

## Chatwoot 的可行性判断

Chatwoot 适合作为第一阶段的多渠道会话中心。

适合的部分：

- 官网聊天
- WhatsApp Business API
- Email
- Instagram / Facebook
- Telegram / Line 等常见渠道
- 销售坐席工作台
- 会话上下文和附件存储
- webhook 通知中间层

需要谨慎评估的部分：

- LinkedIn 私信不是 Chatwoot 稳定原生渠道，可能需要自写连接器
- Slack 更偏内部协作通知，不一定适合作为客户沟通入口
- 特殊渠道可通过 Chatwoot API Channel 接入，但需要开发适配器

结论：

**先用 Chatwoot 做 70%-80% 的会话基础设施，再为特殊渠道写连接器。不要第一阶段从零自研完整客服系统。**

---

## 中间层必须具备的能力

中间层是整个方案的稳定性关键，不能省略。

必须做：

- 接收官网表单
- 接收 Chatwoot webhook
- 校验请求来源
- 消息去重
- 失败重试
- 死信队列或失败日志
- 客户匹配和去重
- 字段映射
- Chatwoot contact/conversation 与 Twenty contact 绑定
- 附件链接保存
- 会话摘要生成
- 定时补偿同步

不能让 Chatwoot 直接裸写 Twenty，否则网络波动、重复 webhook、字段变化时容易出现重复客户、漏消息、错绑定。

## 数据写入 Twenty 的建议字段

建议在 Twenty 中逐步增加这些字段或对象：

### People / 联系人

- 来源渠道
- WhatsApp 号码
- Chatwoot Contact ID
- 最近会话时间
- 最近会话摘要
- 感兴趣产品
- 客户意向等级
- 所属销售
- Primary Channel / 主要沟通渠道
- Chatwoot Conversation URL / 完整会话链接

### ChannelIdentity / 渠道身份（建议新增对象）

- 关联联系人
- 渠道类型：WhatsApp / 官网 chatbot / LinkedIn / Slack / Email / 其他
- 外部用户 ID
- 渠道显示名
- 头像 URL
- Chatwoot Inbox ID
- Chatwoot Contact Inbox ID
- 是否主渠道

### Companies / 公司

- 国家/地区
- 官网
- 行业
- 客户来源
- 关联联系人

### Opportunities / 商机

- 来源：官网表单 / AI 客服 / WhatsApp / 其他
- 感兴趣产品
- 需求描述
- 预计金额
- 阶段
- 负责人

### Notes / 备注

- 会话摘要
- 关键问题
- 客户需求
- 下一步建议
- Chatwoot 完整会话链接

### ConversationLink / 会话索引（建议新增对象）

- 关联联系人
- 渠道类型
- Chatwoot Conversation ID
- Chatwoot Contact ID
- 会话状态
- 当前负责人
- 最后一条消息时间
- 最后消息摘要
- 完整会话 URL
- 是否需要跟进

完整消息和附件仍然建议保存在 Chatwoot；Twenty 保存绑定关系、摘要、索引和前端呈现所需的轻量字段。

## 第一阶段 MVP

目标：先验证链路，不追求一步到位。

1. 部署 Chatwoot 测试环境
2. 接入官网测试聊天窗口
3. 配置 Chatwoot webhook
4. 建一个中间层服务
5. 中间层把 Chatwoot contact/conversation 写入 Twenty
6. Twenty 中能看到客户、来源、最近摘要、Chatwoot 会话链接
7. 先用跳转方式查看完整会话
8. 验证通过后，再做 CRM 内嵌会话面板

验收标准：

- 官网新客户能进入 Chatwoot
- Chatwoot 新会话能同步到 Twenty
- CRM 里能搜到客户和摘要
- 点 CRM 链接能打开 Chatwoot 完整会话
- webhook 重复发送不会重复创建客户

如果直接验证推荐目标方案，则额外验收：

- Twenty 联系人页能显示“多渠道会话”入口
- CRM 页面能拉取 Chatwoot 会话列表
- CRM 页面能查看完整消息上下文
- CRM 页面能展示图片、视频、链接、附件
- CRM 页面能发送一条测试消息，并由 Chatwoot 发出
- Chatwoot webhook 回流后，Twenty 的最后沟通时间和摘要能更新

---

## 第二阶段

目标：加入 AI 智能客服。

1. 建知识库：产品资料、FAQ、公司介绍、报价规则、交期说明等
2. 做 AI Router：判断问题是否能由知识库回答
3. 能回答的问题由 AI 自动回复
4. 不能回答的问题标记并转人工
5. 会话结束后自动总结并写入 Twenty

---

## 第三阶段

目标：扩展更多渠道。

优先级建议：

1. WhatsApp Business API
2. Email
3. Instagram / Facebook
4. Telegram / Line
5. LinkedIn 自定义连接器
6. Slack 视实际场景决定是通知还是客户入口

LinkedIn 和 Slack 不要默认认为能像 WhatsApp 一样直接接入，需要单独验证 API、合规和稳定性。

---

## 主要风险

| 风险 | 说明 | 建议 |
|---|---|---|
| LinkedIn 接入不稳定 | LinkedIn 私信 API 限制较多，不一定适合做自动同步 | 单独做技术验证 |
| WhatsApp 限制 | 官方 Business API 有 24 小时客服窗口和模板消息规则 | 用官方 API，不走个人号灰色方案 |
| CRM 数据膨胀 | 全量消息和附件塞进 CRM 会拖慢系统 | CRM 存摘要和链接，完整记录留 Chatwoot |
| 重复写入 | webhook 可能重复发送 | 中间层必须做幂等 |
| 网络波动 | 写 CRM 失败会造成漏同步 | 中间层必须做队列和重试 |
| AI 误判 | AI 可能错误判断是否需要人工 | 第一阶段保守转人工，逐步优化 |

---

## 当前决策

截至 2026-07-06，推荐方向为：

```text
Chatwoot 作为多渠道聊天中心
Twenty 作为 CRM
中间层作为唯一写 CRM 的同步服务
AI Router + 知识库作为智能客服能力
完整聊天记录保存在 Chatwoot
Twenty 保存客户画像、摘要、标签、来源、会话链接
Twenty 联系人页通过内嵌会话面板呈现聊天记录和回复入口
```

下一步建议：

1. 不改现有 Twenty 数据
